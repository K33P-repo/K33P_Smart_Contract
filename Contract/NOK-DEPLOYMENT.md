# NOK Deployment Guide

Everything the K33P team needs to build, deploy, and integrate the **Next-Of-Kin
(NOK) access registry** Compact contract on Midnight: the ABI, the ZK / secret
key material, the callable circuits, and how a K33P user maps onto the contract.

> The NOK contract lets an **admin** (the K33P backend service) register a hashed
> next-of-kin record for a user, and later approve a NOK-initiated login by
> proving a supplied hash matches the stored record. All hashing is done in code.

## 3. Prerequisites

1. **Compact toolchain** installed (`compact --version` → `0.5.1`).
   Install: <https://docs.midnight.network/develop/tutorial/building/compact>
2. **A proof server** reachable at `PROOF_SERVER_URL` (default `http://127.0.0.1:6300`):
   ```bash
   docker run -p 6300:6300 midnightntwrk/proof-server -- \
     'midnight-proof-server --network preprod'
   ```
3. **A funded wallet** on the target network. Fund via the faucet:
   <https://faucet.preprod.midnight.network/>

## 4. Build

```bash
# from the repo that contains nok-nextofkin/ (workspace root)
npm install

# 1) compile the Compact contract → contract/src/managed/nok (ABI + ZK keys)
npm run compact -w @k33p/nok-contract

# 2) build the TypeScript contract package (so the CLI can import it)
npm run build -w @k33p/nok-contract
```

`npm run compact` produces `contract/src/managed/nok/`:

```
managed/nok/
├── compiler/contract-info.json   # the ABI (see §6)
├── contract/index.{js,d.ts}      # generated TypeScript bindings
├── keys/*.prover  *.verifier     # ZK proving + verifying keys (see §5)
└── zkir/*.zkir *.bzkir           # ZK intermediate representation
```

## 5. Keys & secrets — what they are and who holds them

| Artifact                         | Where                          | Secret? | Purpose |
|----------------------------------|--------------------------------|---------|---------|
| **ZK proving keys** `*.prover`   | `managed/nok/keys/`            | No (public) | Fed to the proof server to build a proof for each circuit call. Ship with the app. |
| **ZK verifying keys** `*.verifier` | `managed/nok/keys/`          | No (public) | Published on-chain at deploy; validators check proofs against them. |
| **Admin secret key** (32 bytes)  | operator machine / K33P vault  | **YES** | The private witness `secretKey()`. Whoever holds it can register/approve. Generated at deploy time; printed once and saved to `nok-deployment.json`. |
| **Admin public key** (32 bytes)  | on-chain `admin` ledger cell   | No | `publicKey(adminSecret)`. Set once in the constructor (sealed). Every write circuit re-derives it from the witnessed secret and compares. |
| **Wallet seed** (hex)            | operator machine / K33P vault  | **YES** | Controls the funding wallet that pays tx fees. |

**Key derivation**
```
adminPublicKey = publicKey(adminSecret)
              = persistentHash([pad(32,"nok:access:pk"), adminSecret])
```
`round` is **not** part of this derivation (fixed in this version), so the admin
identity is stable across every call.

> ⚠️ `nok-deployment.json` contains the `adminSecret`. It is git-ignored. Move it
> into the K33P secret store; never commit it.

## 6. ABI

Source of truth: `contract/src/managed/nok/compiler/contract-info.json`.

### Circuits

| Circuit | Kind | Arguments | Returns | Access |
|---------|------|-----------|---------|--------|
| `publicKey` | pure | `sk: Bytes<32>` | `Bytes<32>` | anyone (client-side only; used to derive the admin key) |
| `register_nok` | proof | `owner_identifier: Field`, `nok_hash: Field` | `[]` | **admin only** |
| `approve_nok_login` | proof | `owner_identifier: Field`, `nok_hash: Field` | `Boolean` | **admin only** |
| `check_nok_registered` | proof | `owner_identifier: Field` | `Boolean` | anyone |

In TypeScript (`bigint` = Field, `Uint8Array` = Bytes):

```ts
Nok.pureCircuits.publicKey(sk: Uint8Array): Uint8Array
deployed.callTx.register_nok(owner_identifier: bigint, nok_hash: bigint): Promise<...>
deployed.callTx.approve_nok_login(owner_identifier: bigint, nok_hash: bigint): Promise<...>
deployed.callTx.check_nok_registered(owner_identifier: bigint): Promise<...>
```

### Witnesses (private state, off-chain)

| Witness | Returns | Provided by |
|---------|---------|-------------|
| `secretKey()` | `Bytes<32>` | `NokPrivateState.secretKey` (the admin secret) |

### Ledger (public on-chain state)

| Field | Type | Meaning |
|-------|------|---------|
| `noks` | `Map<Field, RegisteredNOK>` | registry keyed by `owner_identifier` |
| `admin` | `Bytes<32>` (sealed Cell) | admin public key, set in constructor |
| `round` | `Counter` | monotonic stamp / `registeredAt` |

`RegisteredNOK = { nokHash: Field, ownerIdentifier: Field, registeredAt: Field }`

### Constructor

```
constructor(adminPublicKey: Bytes<32>)   // sealed — set once at deploy time
```

## 7. Deploy

Endpoints come from the shared `cli/src/config.ts` (PreProd by default).

```bash
# PreProd (default)
npm run deploy -w @k33p/nok-cli

# Preview instead
NOK_NETWORK=preview npm run deploy -w @k33p/nok-cli

# Non-interactive (reuse a funded wallet + fixed admin secret)
NOK_WALLET_SEED=<hex> NOK_ADMIN_SECRET=<64-hex> npm run deploy -w @k33p/nok-cli
```

On success it writes `cli/nok-deployment.json`:

```json
{
  "network": "preprod",
  "contractAddress": "0200...",
  "adminPubKey": "….",
  "adminSecret": "….",     // SECRET — move to your vault
  "deployedAt": "2026-…"
}
```

The **contract address** is what K33P stores and uses for all subsequent calls.

## 8. Admin operations — circuits you can call with the contract address

```bash
npm run admin -w @k33p/nok-cli
```

The console loads `nok-deployment.json` for defaults, then offers:

| Menu | Circuit | Inputs (human-readable) | What happens |
|------|---------|-------------------------|--------------|
| 1 Register a next-of-kin | `register_nok` | K33P user handle, NOK identifier | hashes both to Fields, stores the record |
| 2 Approve a NOK login | `approve_nok_login` | K33P user handle, NOK identifier | re-hashes, proves the stored hash matches |
| 3 Check registered | `check_nok_registered` (read) | K33P user handle | membership check against public state |
| 4 View contract state | ledger read | — | admin pubkey, round, record count |

All hashing is automatic (see §9) — the operator types **identifiers**, never hashes.

Programmatic use:

```ts
import { loadConfig } from './config.js';
import { configureNokProviders, joinNok, registerNok, approveNokLogin } from './nok-api.js';
import { ownerIdentifierToField, nokHashToField } from './hash.js';

const providers = await configureNokProviders(walletContext, loadConfig(), logger);
const nok = await joinNok(providers, contractAddress, adminSecret, logger);

await registerNok(nok, ownerIdentifierToField('k33p-user-123'), nokHashToField('+15551234567'), logger);
const ok = await approveNokLogin(nok, ownerIdentifierToField('k33p-user-123'), nokHashToField('+15551234567'), logger);
```

## 9. Automatic hashing

`cli/src/hash.ts` derives all Field values in code (SHA-256, reduced modulo the
BLS12-381 scalar field). Humans never type a hash.

```ts
sha256ToField(input)                 // canonical Field from any string/bytes
ownerIdentifierToField(userHandle)   // domain: "k33p:nok:owner:<handle>"
nokHashToField(nokIdentifier)        // domain: "k33p:nok:hash:<identifier>"
```

Deterministic: the same input always yields the same Field, so `approve_nok_login`
can recompute and compare against what `register_nok` stored.

## 10. K33P integration — mapping a user to `owner_identifier`

`owner_identifier` is an opaque `Field`. K33P derives it deterministically from a
**stable** user handle so the mapping is reproducible without extra bookkeeping:

```
owner_identifier = ownerIdentifierToField(<stable K33P handle>)
```

Recommended handle: the K33P internal user id (immutable). Phone number or wallet
address also work but change over a user's lifetime — prefer the user id.

**Who submits transactions:** the K33P **backend service** holds the admin secret
and the funding wallet, and calls `register_nok` / `approve_nok_login` on the
user's behalf. Suggested flow:

```
K33P DB (user_id) ──ownerIdentifierToField──▶ owner_identifier (Field)
NOK identifier    ──nokHashToField──────────▶ nok_hash (Field)
                    K33P backend (admin secret) ──register_nok──▶ NOK contract
                    NOK login request ─────────▶ approve_nok_login ──▶ allow/deny
```

Store `contractAddress` and the `owner_identifier` alongside the K33P user record;
keep the admin secret and wallet seed in the K33P secret vault.

| # | Issue | Fix |
|---|-------|-----|
| 1 | Contract pinned to obsolete language (`<= 0.18.0`) — won't compile | Bumped to `pragma language_version >= 0.23;`; recompiles on the current toolchain |
| 2 | Admin hardcoded to a placeholder string, compared against a hash → every write aborted | Admin **public key** passed to the sealed constructor; write circuits re-derive it from the witnessed secret; `round` dropped from key derivation |
| 3 | Misspelled / mismatched SDK package names & drifting versions | Correct names, pinned to the compatibility matrix in §2 |
| 4 | Deprecated `testnet-02` endpoints hardcoded in two files | One shared `config.ts` (PreProd default); both `deploy.ts` and `admin.ts` import it |

| 6 | (Documentation only) K33P ↔ `owner_identifier` mapping undocumented | Documented in §10 |

The admin secret is the witness `secretKey()` in `nok.compact`. Every write
circuit re-derives `publicKey(secretKey)` and asserts it equals the on-chain
sealed `admin` cell — so only the holder of this secret can register or approve
NOK records.


The `zkConfigPath` (`cli/src/config.ts`) points here:
`contract/src/managed/nok`. Three circuits each have a `.prover` +
`.verifier` key pair and matching `.zkir` / `.bzkir` intermediate
representations:

- **`register_nok`** — admin registers a hashed next-of-kin record.
- **`approve_nok_login`** — admin approves a NOK-initiated login.
- **`check_nok_registered`** — check whether an owner is registered.

## Redeploying / rotating the admin secret

```bash
# From Contract/
NOK_NETWORK=preview NOK_ADMIN_SECRET=<new-32-byte-hex> npm run deploy
```

If `NOK_ADMIN_SECRET` is omitted, the deploy script generates a fresh secret and
prints it once — save it immediately. Because the `admin` ledger cell is
`sealed` (set only in the constructor), rotating the admin requires a new
deployment.
